\subsection{Begriffe der Modellgetriebenen Software-Entwicklung}
\label{ch:Chapter1:sec:Section1}
% % ===========================

Aus \cite{Truyen2006, Mukerji2001, Miller2001} wird die Terminologie
der modellgetriebenen Software-Entwicklung eingeführt.

\begin{description}
\item[Modell] Ein Modell ist eine formale Spezifikation der
Funktionen, der Struktur und dem Verhalten eines Systems. Das System
wird im Modell in einem bestimmten Kontext und aus einem
bestimmten Sichtpunkt beschrieben. Eine Spezifikation wird formal
genannt wenn sie auf einer Sprache basiert, dessen Elemente eine
wohldefinierte Semantik besitzen. 

%Ein Modell behält die wichtigen Eigenschaften des Modellierten
% Originals. Pragmatismus, Homomorphismus, Abstraktion {Stahl oder
% Stachowiak}?

% Model A model is a formal specification of the function, structure and
% behavior of a system within a given context, and from a specific point
% of view (or reference point). A model is often represented by a
% combination of drawings and text, typically using a formal notation
% such as UML, augmented where appropriate with natural language
% expressions.
% A specification is said to be formal when it is based on a language
% that has a well  defined  semantic  meaning  associated  with  each of
%  its  constructs,  to distinguish  it  from  a  simple  diagram
% showing  boxes  and  lines.  It  is  this formalism, which allows the
% model to be expressed in a format such as XML, in accordance with a
% well-defined schema (XMI).

\item[Modellgetrieben] Unter Modellgetrieben versteht man einen Ansatz
der Software-Entwicklung, worin Modelle als primäres Artefakt für
Dokumentation, Analyse, Design, Bau, Deployement und Wartung dienen.

% Model driven Describes an approach to software development whereby
% models are used as the  primary  source  for  documenting, 
% analyzing, designing,  constructing, deploying and maintaining a
% system.

\item[MDA] Die Model Driven Architecture (MDA) ist ein Ansatz zur
Spezifizierung von Software-Systemen, der die Spezifizierung der
Funktionalität von der Spezifizierung einer konkreten technologischen
Plattform trennt. Die MDA definiert eine Architektur für Modelle, die
eine Menge an Richtlinien zur Verfügung stellt um die in Modellen
ausgedrückten Spezifikationen zu strukturieren. Die Entwicklung der
MDA wird von der Object Management Group (OMG) vorangetrieben.


% The MDA defines an approach to IT system specification that separates
% the specification of system functionality from the specification of
% the implementation of that functionality on a specific technology
% platform. To this end, the MDA defines an architecture for models that
% provides a set of guidelines for structuring specifications expressed
% as models.
% The MDA approach and the standards that support it allow the same
% model specifying system functionality to be realized on multiple
% platforms through auxiliary mapping standards, or through point
% mappings to specific platforms, and allows different applications to
% be integrated by explicitly relating their models, enabling
% integration and interoperability and supporting system evolution as
% platform technologies come and go.

\item[Plattform] Die Plattform ist eine Menge an Untersystemen und
Technologien die Grundaufgaben unterstützende Funktionalitäten
bereitstelen. Implementierungsdetails der Plattform sind für dessen
Klienten irrelevant.

% In the MDA, the term platform is used to refer to technological and
% engineering details that are irrelevant to the fundamental
% functionality of a software component.

% A platform is a set of subsystems and technologies that provide a
% coherent set of  functionality  through  interfaces  and  usage 
% patterns.  Clients  of  a  platform make  use  of  it  without 
% concern  for  its  implementation  details.  Examples  of platforms
% include operating systems, programming languages, databases, user
% interfaces, middleware solutions etc.

\item[Plattformmodell] Das Plattformmodell beschreibt die technischen
Konzepte der Plattformelemente und der zur Verfügung gestellten
Dienstleistungen. Einschränkungen bei der Benutzung der Elemente und
Diestleistungen von anderen Teilen des Systems werden auch
spezifiziert.

% A  platform  model  describes  a  set  of  technical  concepts 
% representing  its constituent elements and the services it provides.
% It also specifies constraints on the use of these elements and
% services by other parts of the system.

\item[Sichtpunkt] Der Sichtpunkt (engl. Viewpoint) ist eine
Abstraktionstechnik um bestimmte Teilaspekte eines Systems näher
beobachten zu können, während alle irrelevanten Details ausgeblendet
werden. Ein Sichtpunkt kann durch ein oder mehrere Modelle
repräsentiert werden. Die MDA spezifiziert drei Standardsichtpuntke
auf ein System: Computation Independent (CI), Platform
Independent (PI), Platform Specific (PS).

% A  viewpoint  is  an  abstraction technique for  focusing on  a 
% particular  set  of concerns within a system while suppressing all
% irrelevant detail. A viewpoint can be represented via one or more
% models.
% MDA viewpoints MDA  specifies  three  default  viewpoints  on  a 
% system:  computation independent, platform independent and a platform
% specific.

\item[CI] Der Computation Independent Sichtpunkt konzentriert sich
auf den Kontext und Anforderungen des Systems, ohne die Stuktur oder die
Verarbeitung in betracht zu nehmen. Das Vokabular des CI Modells gehört der
Domäne an, so dass Domänenexperten und Stakeholder verstanden wird.

% The  computation  independent  viewpoint  focuses  on  the  context 
% and requirements  of  the  system  without  consideration  for  its 
% structure  or processing.

\item[PI] Der plattformunabhängige Sichtpunkt konzentriert sich auf
die operativen Fähigkeiten des Systems ohne eine bestimmte Plattform
anzuvisieren

% The platform independent viewpoint focuses on the operational
% capabilities of a  system  outside  the  context  of  a  specific 
% platform  (or  set  of  platforms)  by showing only those parts of a
% complete specification that can be abstracted out of that platform.

\item[PS] Der plattformspezifische Sichtpunkt erweitert den
plattformunabhängigen Sichtpunkt mit Informationen über die
Verwendung einer bestimmten Plattform.

% A  platform  specific  viewpoint  augments  a  platform  independent 
% viewpoint with details relating to the use of a specific platform.

% \item[Sichttyp] Der Sichttyp (engl. View type)

\item[Modelltransformation] Eine Modelltransformation ist der Prozess in dem ein
Modell in ein anderes Modell überführt wird. Eine Modelltransformation kann ein
Modell mit zusätzlichen Informationen austatten oder es in ein Modell
überführen, dass durch ein anderes Metamodell definiert ist.

\item[Constraint-Sprachen] Eine Constraint-Sprache biete die Möglichkeit Regeln
zu definieren die die Konsistenz der Modellinstanzen sichern. Eine verbreitete
Constraint-Sprache ist die von der UML Standardisierete Object Constraint
Language (OCL).

\end{description}

\subsection{Einsetzbare Technologien}
\label{ch:Chapter1:sec:Section1}
% % ===========================

Unter den in der modellgetriebenen Entwicklung eingesetzten Technologien, um
Transformatioen zu unterstützen, zählen unter anderen auch QVT-O und ATL. Für
die Erstellung und Handhabung von Domänenspezifischen Sprachen bietet sich das
Xtext Projekt an und Modell zu Text Traksformationen werden von dem XPand
Projekt unterstützt. Die oben erwähnten Technologien sind Teil des Eclipse
Modelling Projekts und genießen eine starke Unterstützung durch die
Entwickler-Community\footnote{http://www.eclipse.org/modeling/}.

Die Epsilon Plattform\footnote{http://www.eclipse.org/epsilon/} bietet eine
Alternative zu den oben genannten Technologien, kann jedoch auch in die Ecplise
Umgebung eingebunden werden. Epsilon ist eine erweiterbare Plattform, die
Unterstützung für die Erstellung und Interoperabilität von Domänenspezifischen
Sprachen und für Modell-Management Aufgaben wie Transformation,
Code-Generierung, Vergleich, Validierung und Veränderung von Modellen.

\subsection{Begriffe des Produktlinien-Engineering}
\label{ch:Chapter1:sec:Section1}
% % ===========================

Um den Begriff des \emph{Software-Product-Line-Engineering} (SPLE) zu
erklären werden zuerst die Begriffe der Hauptbestandteile erläutert.
In \cite{Pohl2005} wird \emph{Mass customisation} (kundenindividuelle
Massenproduktion) als die Großproduktion von Gütern die auf die
individuellen Bedürfnisse der Kunden angepasst wurde definiert. Eine
\emph{Plattform} ist eine Technologiebasis auf der andere Technologien
oder Prozesse aufbauen. SPLE ist ein Paradigma um Software-Anwendungen
mit Hilfe von Plattformen und kundenindividueller Massenproduktion zu
erstellen.\\
Um die individualisierte Massenproduktion [massenanfertigung] zu
ermöglichen werden Artefakte benötigt die an die verschiedenen Systeme
der Produktlinie angepasst werden können. Diese Flexibilität der
Artefakte wird \emph{Variabilität} genannt. Die Aktivität, die als
Ziel die Variabilität zu identifizieren und zu dokumentieren hat,
heißt \emph{Variabilitätsdefinition}. Im Prozess des
Produktmanagements werden gemeinsame und variable Merkmale  der
Software-Entwicklungslinie identifiziert. Die Modellierung der
Variabilität unterstützt die Entwicklung und die Wiederverwendung der
Variablen Artefakte. Die Variabilität der Domänenanforderungen,
-Architektur, -Komponenten und -Tests wird mit Hilfe des
Variabilitätsmodells festgehalten.\\
Ein \emph{Entscheidungsmodell }(Decision Model) beinhaltet die Beziehungen
zwischen den Elementen einer Entscheidung und den prognostizierten
Resultaten um Problemstellungen verstehen und kontrollieren zu können.
Somit können die Resultate verschiedener Handlungen vorhergesagt
werden \cite{DecModel}.

Weitere Begriffe die im Zusammenhang mit SPLE verwendet werden
\cite{Pohl2005}:
\begin{description}   
 \item[Variabilitässubjekt] Ein Variabilitätssubjekt (Variability
 Subject) ist ein variabler Gegenstand der reelen Welt oder eine
 variable Eigenschaft eines solchen Gegenstands.
 \item[Variabilitäsobjekt] Ein Variabilitätsobjekt (Variability
 Object) ist eine Instanz eines bestimmten Variabilitätssubjekts.
 \item[Variationspunkt] Der Variationspunkt (Variation Point) ist
 eine Repräsentation des Variabilitätssubjektes innerhalb der
 Domänenartefakte, das mit zusätzlichen Kontextinformationen
 angereichert wurde.
 \item[Variante] Eine Variante ist die Repräsentation eines
 Variabilitätsobjektes innerhalb der Domänenartefakte.
 \item[Interne Variabilität] ist die Variabilität eines
 Domänenartefaktes, das vor den Kunden versteckt ist.
 \item[Externe Variabilität] ist die Variabilität eines
 Domänenartefaktes, das von den Kunden sichtbar ist.
\end{description}

\section{DocBook}
\label{sec:docBook}
% ==================

DocBook ist eine Sammlung von Standards und Werkzeugen für Technische
Publikationen. Der Standard wird von dem technischen Komitee OASIS
(Organization for the Advancement of Structured Information Standards)
\cite{DocBook} gepflegt und wurde von einem Konsortium von
Software-Firmen erstellt um Computer-Dokumentation zu standardisieren.

Das Vokabular und der Inhalt der Elemente die ein Autor benutzen kann
und  die Verbindungen zwischen den Elementen wird durch das DockBook
DTD (Document Type Definition) definiert. Das Format eignet sich
besonders in folgenden Situationen:
\begin{itemize}
  \item Sehr große Mengen an Inhalten
  
  \item Sehr strukturierte Inhalte 
  
  \item Inhalte die zwischen inkompatiblen Systemen
  ausgetauscht werden müssen
  
  \item Inhalte die automatisierete Batch-Verarbeitung
  benötigen
  
  \item Inhalte die in verschienenden Ausgabeversionen und
  -formen dargestellt werden müssen
\end{itemize}

Dadurch dass DocBook eine semantische Sprache ist, können Inhalte
in einer präsentationsneutralen Form die ihre logische Struktur
erfasst, dargestellt werden.

Das Markup ist ähnlich HTML mit Elementen die Bücher, Titel,
Paragraphen oder Kapitel repräsentieren. Stylesheets die es erlauben
die Inhalte in verschieden Formaten wiederzugeben sind auf
\cite{DocBookSF} frei zugänglich. Wie in \cite{Stayton2007} erwähnt
wird gibt es viele Textverarbeitungssysteme die DocBook unterstützen.
Für kleine, einmalige Dokumente wird das System nicht empfohlen. Es
ist auch für Dokumente die stark Layout-getrieben sind nicht geeignet.
Da die Dateien in Klartext abgespeichert werden und nicht binär,
können Versionskontrollsysteme bentuzt werden.
